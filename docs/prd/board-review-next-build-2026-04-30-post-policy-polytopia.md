# Purpose

This memo records the board-guided lean-coffee answer to:

What should Liftoff focus on next after:

- canonical buyer scorecards
- buyer attribution audit
- opportunity mix
- explore vs exploit
- constrained-surface exposure
- throughput and follow-through
- allocation trigger policy

using an `Elon Musk / Polytopia` lens:

- capture the highest-leverage territory first
- simplify to the fewest compounding control surfaces
- build the machines that build the machines
- optimize for learning-rate and expansion-speed, not ornamental completeness

# Prime Directive

Maximize durable net profit growth per unit of constrained company capacity, while never allowing projected monthly net profit to fall below the configured floor.

# Musk / Polytopia Interpretation

In practical operating-system terms, the Polytopia lens means:

1. claim the tiles that compound everything else
2. do not spread engineering effort evenly across the map
3. build the shortest loop from signal to action to learning
4. remove human interpretation layers where the signal is already good enough
5. preserve scarce surfaces while expanding onto new profitable territory

So the next board question is not:

`what other data can we show?`

It is:

`what control surface most increases expansion speed without blowing up scarce account, operator, or buyer capacity?`

# Current Built State

The system can now tell:

- who the buyer probably is
- how they are performing economically
- what mix they carry
- whether they are exploring or exploiting
- whether constrained surfaces are distorting the read
- whether follow-through is real
- what allocation posture and trigger policy follows

That means the next bottleneck is no longer visibility.

The next bottleneck is:

`conversion of grounded policy into faster operating action and faster learning`

# First-Principles Read

The company now has three core scarce resources:

1. safe buying surfaces
2. buyer attention / execution bandwidth
3. upstream opportunity supply

The highest-leverage moves are the ones that:

- preserve `1`
- route `2` toward the best opportunities
- increase the quality and speed of `3`

Anything else is lower-order.

# Lean Coffee Order

## 1. Allocation Execution Engine

### Why It Needs To Exist

The system can now say:

- block incremental spend
- hold current allocation
- allow measured growth
- allow scale

But it still does not turn that policy into an executable operator queue.

Right now policy is visible but still manually interpreted.

### Why Now

The trigger-policy layer was the last major truth prerequisite.

Once policy is credible, the next win is to shorten the loop from:

`buyer state -> allocation decision -> concrete next action`

### Limiting Factor

`allocation policy exists, but it is not yet an execution engine`

### Responsible Board Seats

- `Elon Musk`
- `Jim Simons`
- `Patrick Collison`

### Musk / Polytopia Rationale

This is a city-upgrade move, not a scouting move.

It takes an already-claimed tile and makes it produce more per turn.

## 2. Packet Lineage And Reuse Graph

### Why It Needs To Exist

The system still cannot cleanly distinguish:

- true invention
- self-reuse
- peer-copying
- derivative packet evolution

That means buyer scorecards still partially blur skill, originality, and exploit behavior.

### Why Now

Explore vs exploit is already attached, but the exploit side is still too coarse.

The next truth upgrade is lineage:

who created what, who copied what, and what actually propagated.

### Limiting Factor

`the system still cannot fully separate original edge creation from derivative reuse`

### Responsible Board Seats

- `Founding Partner Seat`
- `Jeff Bezos`
- `Jensen Huang`

### Musk / Polytopia Rationale

This is map intelligence.

It tells us whether we are discovering new territory or merely marching around on already-owned tiles.

## 3. Surface Preservation And Recovery Command Layer

### Why It Needs To Exist

Safe buying surfaces are currently one of the hardest constraints in the system.

We can see surface risk, but we do not yet have a dedicated command layer for:

- recovery state
- restriction chronology
- appeals
- keepalive tasks
- redirect rotation state
- sanctioned automation readiness

### Why Now

The allocator now explicitly uses surface constraints, which means the next compounding move is to improve the surfaces themselves.

### Limiting Factor

`surface preservation is visible, but not yet run as its own high-discipline command loop`

### Responsible Board Seats

- `Mark Zuckerberg`
- `Elon Musk`
- `Warren Buffett`

### Musk / Polytopia Rationale

This is border defense.

If we lose high-value cities, expansion speed elsewhere becomes irrelevant.

## 4. Buyer Daily Command Packet

### Why It Needs To Exist

The system knows enough now that each buyer should receive a concise operating packet:

- current posture
- constrained surfaces
- owned stale opportunities
- expected next moves
- what is blocked
- what should be explored next

Without this, the operating system still requires too much operator mediation.

### Why Now

Once allocation policy exists, the next fastest leverage is to put the right packet in front of each buyer every day.

### Limiting Factor

`the system knows what matters, but it does not yet deliver buyer-specific daily operating instructions`

### Responsible Board Seats

- `Jeff Bezos`
- `Patrick Collison`
- `Founding Partner Seat`

### Musk / Polytopia Rationale

This is factory throughput.

It reduces idle thinking time and turns the command system into actual motion.

## 5. Opportunity Supply Quality Loop

### Why It Needs To Exist

The upstream opportunity system now has ownership, but it still needs a tighter quality loop around:

- what sources actually convert
- what categories produce durable margin
- what exploratory work produces scale-worthy packets
- where sniffing quality is weak

### Why Now

Once the allocator and buyers are better directed, the next constraint becomes the quality of new territory being discovered.

### Limiting Factor

`opportunity supply exists, but the quality-improvement loop is still too manual and too diffuse`

### Responsible Board Seats

- `Founding Partner Seat`
- `Warren Buffett`
- `Jim Simons`

### Musk / Polytopia Rationale

This is expansion velocity.

The empire grows only if new tiles are worth owning.

# Requirements To Delete Or Relax

From the Musk seat, the board should explicitly delete or relax these weak requirements:

1. `Every next step needs a new data surface`
   The system now has enough truth to act on several loops directly.

2. `Allocator output can stay advisory for now`
   False. Once policy is credible, advisory-only mode becomes avoidable drag.

3. `Buyer management can remain operator-mediated`
   False. The system should increasingly push daily command packets outward.
