# Overnight Sprint Contract

## Purpose

This contract pre-authorizes a sequence of tightly scoped overnight operating-system sprints so the system can keep compounding without stopping to renegotiate the build frame every few hours.

The intent is not to widen scope.

The intent is to run the same disciplined loop repeatedly:

1. board-led lean coffee
2. one dominant bottleneck
3. `4-5` hours of focused build time
4. durable review packet
5. operator checkpoint before broadening again

## Governing Rule

Each sprint is a bounded build block, not an open-ended roadmap grant.

Each sprint must still satisfy:

- prime-directive alignment
- first-principles rigor
- board-guided prioritization
- explicit checkpoint review

This contract does **not** override the existing guardrails in:

- [operator-delegated-approval-protocol.md](/Users/ericroach/code/liftoff/docs/prd/operator-delegated-approval-protocol.md)

## Lead Board Member

Primary sprint lead:

- `Elon Musk`

Why:

The sprint frame is about compounding control loops, shrinking operator drag, and turning the system from an intelligence surface into a machine that actually runs the business rhythm.

## Lean Coffee Operating Pattern

Each sprint should be proposed the same way:

1. name the bottleneck plainly
2. identify the first real unlock
3. order the build slices by leverage
4. keep only one dominant theme in the sprint
5. end with a simple operator pitch in under three minutes

## Sprint Sequence

## Sprint 01

- `name`: `Operator Autopilot Readiness`
- `status`: `current reference sprint`
- `duration`: `4-5 hours`

### Sprint Goal

Close the loop from:

`recommendation -> action -> outcome -> improved next recommendation`

### Core Deliverable

By the end of the sprint, the operating system should be able to run the morning control ritual in one place with minimal operator stitching.

### Lean Coffee Order

1. `Command Outcome Telemetry`
2. `Morning Operator Packet Assembly`
3. `Auto-Rollup Of State Changes`
4. `Escalation Rules`
5. `Buyer-Facing Readiness Check`

### Definition Of Done

The operator can open one surface and answer:

1. what matters first
2. what changed since yesterday
3. what got unstuck
4. what is still blocked
5. what is now ready for promotion or growth

### Reality Note

This sprint is already in progress and has already produced the foundational queue/control work. It should be treated as the reference sprint template for the two follow-on sprints below.

## Sprint 02

- `name`: `Morning Packet And Outcome Closure`
- `status`: `pre-approved after Sprint 01 checkpoint`
- `duration`: `4-5 hours`

### Sprint Goal

Turn the unified operator queue into a morning operating packet that automatically explains:

- what changed
- what moved state
- what improved posture
- what remains blocked

### Lean Coffee Order

1. `Command Outcome Telemetry`
2. `Morning Operator Packet Assembly`
3. `Auto-Rollup Of State Changes`
4. `Escalation Rules`
5. `Yesterday-To-Today Delta Summary`

### First Unlock

The system stops just ranking commands and starts proving whether operator action changed anything.

### Definition Of Done

The operator receives or opens one packet and can immediately see:

1. the top queue today
2. the state transitions since the last cycle
3. the commands that cleared or promoted
4. the commands that remained stuck
5. the most important escalation requiring intervention

## Sprint 03

- `name`: `Buyer Readiness And Controlled Delegation`
- `status`: `pre-approved after Sprint 02 checkpoint`
- `duration`: `4-5 hours`

### Sprint Goal

Decide which operator commands are clean enough to eventually become buyer-facing without actually turning on outbound automation yet.

### Lean Coffee Order

1. `Buyer-Facing Readiness Check`
2. `Command Cleanliness Rules`
3. `No-Send Preview Layer`
4. `Escalation-To-Buyer Boundary`
5. `Operator Override Rules`

### First Unlock

The system can distinguish between:

- commands safe for internal operator handling only
- commands clean enough for future outward delegation

### Definition Of Done

The operator can tell, for any command:

1. whether it is internal-only
2. whether it is buyer-ready
3. why it is not yet safe to delegate outward
4. what must change before buyer-facing automation is ever allowed

## Checkpoint Rule

Each sprint ends with a short checkpoint packet that answers:

1. what was built
2. what bottleneck it relieved
3. what seems to be landing
4. what still feels weak
5. what the next sprint should inherit

## Non-Goals

These overnight sprints are **not** for:

1. random dashboard expansion
2. broad repo cleanup unrelated to the sprint bottleneck
3. turning on outbound buyer messaging
4. live spend or traffic changes without explicit operator approval
5. speculative side projects outside the current control-loop theme

## Simplest Pitch

The sprint program is trying to achieve one thing:

`make the operating system capable of running the morning control loop with less human stitching and more proof that action changed outcomes`

That is the contract for the next two sprints after the current reference sprint.
